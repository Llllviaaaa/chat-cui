# Phase 01.1: PC Agent plugin architecture alignment with message-bridge-opencode-plugin - Context

**Gathered:** 2026-03-03
**Status:** Ready for planning

<domain>
## Phase Boundary

This urgent phase corrects architecture direction: PC Agent must be implemented as a plugin of the enterprise collaboration client, and its core responsibility is bridging local OpenCode with AI-Gateway using the internal protocol boundary.

Scope is HOW to realize this plugin-aligned bridge path and migration from the current Java-oriented path. It does not add new user-facing business capabilities beyond this architecture correction.

</domain>

<decisions>
## Implementation Decisions

### Plugin Identity and Core Responsibility
- PC Agent is a client plugin in the office collaboration software, not an independent Java client application.
- Core responsibility is fixed: connect local OpenCode and AI-Gateway, and perform bidirectional protocol bridging.
- This definition is a hard constraint for phase completion: any architecture drift is considered not done.

### Alignment Baseline with Reference Repository
- Alignment target is structure + behavior dual alignment against `message-bridge-opencode-plugin` reference style.
- Project topology should mirror layered plugin structure (core/bridge/host-adapter style separation).
- Lifecycle contract must be complete: init/start/stop/dispose + reconnect state machine + error/health events.
- Host-client to plugin interaction should be event-driven contract.
- Protocol conversion ownership is fully on plugin layer (OpenCode <-> internal protocol).
- Post-alignment evolution policy: use internal versioning (no mandatory continuous upstream sync).

### Runtime Modes and Delivery Gate
- Phase 01.1 must deliver both modes:
  - Plugin mode (embedded in client)
  - Standalone debug mode (without client)
- Standalone mode form is CLI.
- Standalone mode must use real chain (real OpenCode + real AI-Gateway), not mock-only path.
- Acceptance requires reproducible end-to-end evidence for both modes.

### Migration and Cutover Policy
- Existing Java `pc-agent` runtime path is to be replaced in this phase (one-shot cutover preference).
- Legacy Java code handling: archive migration references, then remove from main path.
- Existing high-value test intent should be ported into plugin-side contract/integration tests.
- Credential handling must preserve existing security semantics (OS secure storage / DPAPI-equivalent constraints).
- Gateway auth/protocol contract must remain strictly compatible with current AUTH/gateway boundary in 01.1.
- Rollback strategy decision: no dedicated rollback path for this phase.

### Claude's Discretion
- Concrete package/module names under the aligned layered topology.
- Event naming and payload schema details under the event-driven contract.
- CLI debug mode UX details (command names, flags, output formatting) as long as dual-mode and real-chain constraints are satisfied.

</decisions>

<specifics>
## Specific Ideas

- "PC AGENT is a plugin used to connect gateway server and OpenCode." 
- The plugin belongs to the office collaboration client ecosystem, not a standalone product path.
- Even without the client, there must be an independent joint-debug mode to connect service side and OpenCode.
- Completion standard is strict: deviation from the plugin-bridge definition is treated as incomplete.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `gateway` module already provides AUTH_V1 auth boundary, deterministic error contract, HTTP/WS interceptors, and related tests.
- Current `pc-agent` Java module contains reusable security semantics reference:
  - auth config loading and credential state checks
  - Windows secure credential storage abstraction and typed failure handling

### Established Patterns
- Security/auth boundary is already established in Phase 1 and should be preserved in 01.1.
- Existing repository currently has Java Maven modules (`gateway`, `pc-agent`) and no Electron/TS plugin scaffold yet.

### Integration Points
- Client plugin host lifecycle/events integration point in desktop collaboration client runtime.
- AI-Gateway auth + connection establishment integration point (existing gateway contracts).
- Local OpenCode runtime bridge integration point (plugin-side protocol conversion ownership).
- Standalone CLI debug entry point must exercise the same bridge core as plugin mode.

</code_context>
